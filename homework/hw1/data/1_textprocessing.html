Overview of basic text preparation steps using quanteda

In practical applications we will want to bring in a collected corpus into quanteda for analysis.

Assume that you have scraped English language articles from two North Korean news outlets for the year 2015. Your corpus consists of articles from: the Korean Central News Agency (KCNA) and the Rodong Sinmun (ROSI), which is the official newspaper of the Central Committee of the Workers Party of Korea. You have written the text and document-level variables to disk as a comma-separated values (.csv) file.

Below is code which will take you through the basic text preparation and processing workflow.

Bringing in the text

Below is code which will allow you to construct a quanteda corpus using external data using the readtext function. We will be using this text data for the remainder of the tutorial.

Our first task is to bring in the text and metadata (also called docvars in quanteda terminology) into a format that will serve as a reference corpus object will serve all downstream applications. Below we bring in the data usine readtext, create a corpus object using the corpus function, and then display the summary stats for the first 3 documents in the corpus object.

Text Processing

Once we have pulled in our text and created a corpus, the next step is to process the text prior to generating a document-feature matrix. In the code below, we will process the text step by step using the tokens function to create a tokens object, while observing the changes made at each stage of the pipeline.

The demonstration will proceed as follows:

  1. Tokenize the documents
  2. Lowercasing
  3. Removal of numbers, punctuation, hyphens, symbols, separators, and URLs.
  4. Stopword removal
  5. Stemming

By stemming, we are able to collapse multiple forms of the same term onto a single type by reducing the inflections of words and collapsing onto the base form of the word. We can see, however, that some tokens might become difficult to interpret and that different words can merge together erroneously because they happen to have a common base form. We should always be cognizant of these trade-offs when deciding to stem.

Collocations and compounding multi-word expressions

There are usually terms in our corpus that are multi-word expressions (collocations) and should be treated as individual tokens in our analysis downstream. Sometimes, we have theoretical priors on what these terms are. For instance, in the current corpus, we can be fairly confident that the term Kim Jong Un will show up. On the other hand, there may be terms that are frequent but that we cannot anticipate. In this case, it would be useful to employ an unsupervised collocation detection method, and then compound the constituent words in the collocation based on the results of this analysis.

We can compound the tokens in our corpus by creating and manipulating a tokens object from our corpus.

First, lets do some exploration. How are some well-known multi-word expressions used in the corpus?

  1. Create a tokens object from our quanteda corpus object
  2. Ensure that the tokens object does not have stopwords, that the tokens are lowercased, and we will also remove punctuation and separators with regular expressions.
  3. Then we will perform some KWIC commands to inspect usage of known collocations. Note that we set the padding argument to true in the tokens_remove function so as to make sure that there is a whitespace in place of a removed stopword. This will ensure proper positional matching when detecting the collocations.
toks_kcna <- tokens(corpus_kcna,
                    include_docvars = TRUE) %>%
  tokens_tolower() %>%
  tokens_remove(stopwords('english'), padding = TRUE) %>%
  tokens_remove('[\\p{P}\\p{S}]', valuetype = 'regex', padding = TRUE)

leader_col <- kwic(toks_kcna, pattern = phrase(c('kim jong un', 'kim jong il', 'kim il sung', 'supreme leader')))

toks_kcna_col <- tokens_compound(toks_kcna, 
                                 pattern = leader_phrases,
                                 concatenator = "_",

Construction of document-feature matrix and feature selection

Having processed the text, we will now generate of bag of words representation of the corpus. That is, we will construct a document-feature matrix. Then, we will think about how we can further reduce the dimensionality of the matrix without losing crucial information.

To create a document-feature matrix (DFM), we can use the dfm function in quanteda. This function accepts character vectors, corpus objects, tokens objects, and other dfm objects. Take a look at the documentation, as there are really fast ways to convert a corpus into a DFM. For instance, we could construct a DFM while performing all the cleaning steps we did above as follows:

dfm_kcna_stem <- corpus_kcna %>%
  quanteda::tokens(remove_numbers = TRUE,
                   remove_punct = TRUE,
                   remove_symbols = TRUE,
                   remove_hyphens = TRUE,
                   remove_separators = TRUE,
                   remove_url = TRUE,
                   include_docvars = TRUE) %>%
  tokens_wordstem() %>%
  dfm() # clean and stem

dfm_kcna_stem_nostops <- dfm_select(dfm_kcna_stem, pattern = stopwords("en"), selection = "remove") # remove stop words on cleaned dfm

For our purposes, I will bring in the toks_kcna_col tokens object that contains the collocations we replaced before and which does not have stemmed tokens. Note that we could simply pass the argument stem = TRUE at this point to stem the tokens if we so wished.

dfm_kcna <- dfm(toks_kcna_col) # create a DFM

Further dimensionality reducion of the DFM

A DFM of a large corpus such as ours is likely to have a significant number of features that appear sporadically. We might want to remove very rare features from our DFM so that the dimensionality of the matrix can be reduced further. By doing so, we can reduce computational costs during the analysis stage. This might also help discard generally irrelevant features.

To demonstrate this, lets only keep features that appear at least, say, 10 times in the corpus. We can accomplish this with trim.

dfm_kcna_trim <- dfm_trim(dfm_kcna,
                          min_termfreq = 10,
                          verbose = TRUE)
## Removing features occurring:

By reducing the number of features by either using a threshold of minimum term frequency or minimum document frequency (number of documents the term appears in), we can help boost computational efficiency. Of course, trimming comes at the expense of removing features that could be informative. It is important to be sensitive to this trade-off.
